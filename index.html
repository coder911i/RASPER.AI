What is ResPar?

ResPar (Resume Parser) is an AI-powered recruitment automation system built entirely on Google's free infrastructure. It automatically processes job applications submitted via Google Forms, extracts resume content from Google Drive, runs AI analysis against job descriptions, and displays ranked results on a live web dashboard — all without any paid hosting or servers.

The system is designed for small HR teams or solo recruiters who receive resumes via Google Forms and want instant AI scoring without expensive ATS software. Everything lives inside Google Sheets, Google Drive, and Google Apps Script.

Zero cost to run. Uses Groq's free tier (14,400 requests/day), Google Apps Script (free), and Google Sheets (free). No AWS, no databases, no monthly fees.

How It Works — End to End

Form Submit
Candidate fills Google Form with name, email, resume link, JD choice
⚡

onFormSubmit() runs automatically via Apps Script trigger
📄
Resume Read
4-method fallback extracts text from Google Drive file
🤖
AI Analysis
Groq (Llama 3.1) scores resume vs JD and returns JSON
💾
Save & Alert
Results saved to sheet, email sent if score ≥ 80%
📊
Dashboard
Live web app shows ranked candidates, auto-refreshes every 15s

The system uses a 4-method resume extraction fallback chain: it first tries Google Doc text export via API, then direct DocumentApp read, then file copy, then raw blob parsing. This ensures maximum compatibility with different file formats submitted by candidates.

The AI prompt is carefully structured to always return a strict JSON object with a match score (0–100), matched skills, missing skills, experience level, strengths, concerns, recommendation, and a summary. This structured output makes it trivial to store, sort, and display results.

How Tanishq Built It
T
Tanishq
Built ResPar end-to-end — backend automation, AI integration, and the live dashboard — using Google Apps Script as the sole infrastructure layer, keeping the entire system free to run.

The system was built in several stages. First, the Google Sheets structure was set up with three sheets: Candidates (linked to a Google Form), Job Descriptions (editable JD library), and Analysis Results (where AI output is written).

Next, the backend automation was written in Apps Script. The core loop is: read form row → extract resume text from Drive → call Groq AI → parse JSON response → write results to sheet → send email if high score. The resume extraction required a 4-method fallback chain to handle different Drive file types reliably.

The Groq AI integration was set up using the OpenAI-compatible API endpoint. The prompt was designed to always return a strict JSON object, using Groq's response_format: json_object parameter to enforce this. The API key is stored securely in Script Properties, not hardcoded.

Finally, the frontend dashboard was built as a single HTML file served by Apps Script's doGet() function. It uses google.script.run to call backend functions directly from the browser, with a 15-second auto-refresh loop and toast notifications for live updates.

Spreadsheet Structure

The Google Spreadsheet has 3 sheets. Here's what each one contains:

1
Candidates
Linked to Google Form. Columns: Timestamp, Name, Email, Resume Link (Google Drive URL), JD Name (e.g. JD1/JD2), Status (blank → PROCESSING → DONE/ERROR). This is the input sheet — do not edit manually.
2
Job Descriptions
Two columns: JD Name (e.g. JD1, JD2, JD3) and Job Description (full text of the role requirements). Add new rows here to support new positions. The candidate selects which JD to be scored against at form submission.
3
Analysis Results
Written automatically by the backend. Columns: Timestamp, Name, Email, JD Used, Match Score (colour-coded), Skills Matched, Skills Missing, Experience Level, Strengths, Concerns, Recommendation, Summary. This feeds the dashboard.

How to Test the System

Follow these steps to do a complete end-to-end test of ResPar:

1
Verify Setup
In Apps Script editor, run healthCheck(). Confirm SPREADSHEET_ID and GROQ_API_KEY both show ✅ in the execution log.
2
Prepare a Test Resume
Create a Google Doc with some sample resume text (name, skills, experience). Make sure it's set to "Anyone with the link can view". Copy the sharing URL.
3
Submit the Google Form
Open the Google Form linked to the Candidates sheet. Fill in: Name, Email, paste the resume Drive link, and select a JD (JD1, JD2, or JD3). Submit.
4
Watch the Candidates Sheet
Open the Candidates sheet. Within a few seconds the Status column should change from blank → PROCESSING... → DONE. If it shows ERROR, check Apps Script execution logs.
5
Check Analysis Results Sheet
Switch to the Analysis Results sheet. A new row should appear with the AI score, matched skills, missing skills, recommendation, and summary. The score cell will be colour-coded.
6
Open the Dashboard
Go to Deploy → Manage Deployments → copy the Web App URL → open in browser. The candidate should appear with their score ring, skills, and expandable detail card. Click the card to expand strengths, concerns, and missing skills.
7
Test Manual Processing
To reprocess all pending rows manually (e.g. after setup), run processAllPending() from the Apps Script editor. Useful if the form trigger wasn't set up before submissions came in.
Tip: If a candidate scores 80% or higher, you'll also receive an automatic email alert with their name, score, recommendation, and AI summary. Check your inbox after submitting a strong resume.

Backend Key Functions Reference
GS
setupResPar()
Run once after deployment. Saves SPREADSHEET_ID and GROQ_API_KEY to Script Properties. Creates the Analysis Results sheet if missing.
GS
onFormSubmit(e)
Trigger function. Fires on every Google Form submission. Waits 3 seconds then calls processCandidate() for the new row.
GS
processCandidate(rowNumber)
Main processing function. Reads candidate row, extracts resume, calls Groq AI, saves results, updates status to DONE or ERROR.
GS
analyzeWithGroq(resumeText, jdText, candidate)
Calls Groq API with structured prompt. Returns parsed JSON with matchScore, skillsMatched, skillsMissing, experienceLevel, strengths, concerns, recommendation, summary.
GS
getAllResults()
Called by the dashboard UI via google.script.run. Returns all rows from Analysis Results sheet as an array of objects keyed by column header.
GS
processAllPending()
Manual batch function. Loops through all Candidates rows and processes any with blank or ERROR status. Useful for initial bulk processing.
GS
healthCheck()
Debug helper. Logs SPREADSHEET_ID, GROQ_API_KEY status, sheet names, and row counts. Run this first if anything seems broken.

Known Limitations

Resume format: Best results with Google Docs resumes. PDF extraction is limited — the blob method often produces garbled text from PDFs. For best accuracy, ask candidates to submit Google Doc links.

Groq rate limits: Free tier allows 14,400 requests/day and ~30 requests/minute. High volume bursts may hit rate limits. The system retries once after a 429 error but doesn't queue.

Apps Script execution time: Maximum 6 minutes per execution. Processing a single candidate typically takes 5–15 seconds. Batch processing many candidates at once may time out.

Concurrent submissions: If multiple forms are submitted simultaneously, triggers may overlap. The PROCESSING status check prevents double-processing in most cases.

Roadmap — What Could Be Added

Two features have been identified as strong candidates for a future version of ResPar:

Interview Question Generator — for each candidate, automatically generate tailored interview questions based on their skill gaps, strengths, and the JD they applied for. These would appear directly on the candidate card in the dashboard.

Candidate Comparison View — select any two candidates and compare them side-by-side with a visual score and skills breakdown, helping recruiters make faster shortlisting decisions.

Both would work entirely with data already in the system — no new sheets, forms, or infrastructure needed.
